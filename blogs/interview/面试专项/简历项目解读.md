## 对平台进行i18n国际化语言配置，解决国际化词条动态拼接、增量i18n提效等痛点

基于**i18next**体系，接入了腾讯**ti18n**平台，ti18n在i18next的基础上封装了CDN加载和CLI提取能力，实现翻译资源CDN化管理（翻译发布和代码解耦，翻译更新不用走CI/CD）。

工程化：ti18n CLI扫描源码，自动生成待翻译列表，防止遗漏

翻译管理：平台化写作，翻译人员无需接触代码仓库

降级策略：key未找到的时候显示原文

### 亮点

1、语言切换的时候会刷一下页面

“是的，语言切换时我们采用了主动触发页面重刷（Reload）的策略。这看似传统，但在我们当时 100K+ 语言包大小 的特定场景下，是最优的架构折中方案。

首先，如果采用单页应用（SPA）内的动态响应式切换，这意味着我们必须在内存中同时维护或动态异步加载多套 100K+ 的语言包，并且会触发整个 React/Vue 组件树的大面积重新渲染，容易造成明显的卡顿和内存占用。

其次，平台中很多第三方插件、底层图表、甚至部分硬编码数据是在全局初始化时一次性注入的，SPA 内热更新很难完全覆盖这些死角。

因此，我们选择在用户切换语言时，**更新本地存储（LocalStorage/Cookie）的语言标记并直接 Reload 页面**。配合 **CDN 的极速加载**，浏览器会重新请求对应语种的单一 JSON 文件（100K+ 压缩后其实很小）。这样做不仅**保证了所有第三方库和组件能 100% 干净、彻底地完成语言初始化**，而且避免了复杂的响应式状态维护，对团队来说是一种更低心智负担、更稳健的生产环境实践。”

2、翻译资源 CDN 化管理，不走 CI/CD，是怎么保证线上安全的？（降级策略）

- **完美作答**： “解耦 CI/CD 意味着运营或翻译人员在 ti18n 平台一键发布，线上就能实时生效。这带来了极高的效率，但也带来了风险（比如翻译人员不小心删除了某个 Key）。 为了闭环这个风险，我们建立了**多重降级与容灾策略**：
    
    1. **代码级兜底（Key未找到显示原文）**：在 i18next 初始化时配置了 `fallbackLng` 和自定义的 `saveMissing` 逻辑。一旦 CDN 的 JSON 里某个新词条由于时差没同步成功、或者被误删，系统**绝对不会抛错崩溃或显示空白**，而是优雅地降级显示代码里的 `Key` 本身（或默认兜底语言的原文），确保基础链路可用。
        
    2. **CDN 缓存版本控制**：ti18n 平台生成的 CDN 带有版本号或时间戳，我们在前端加载时会做合理的失效时间控制（Stale-While-Revalidate 策略），防止因 CDN 节点刷新延迟导致的用户语言包不一致问题。”

3、ti18n CLI 扫描源码，动态拼接的词条它是怎么识别出来的？

- **完美作答**： “CLI 主要是基于 **AST（抽象语法树）** 匹配指定的标识符（如 `t('...')`）。对于**动态拼接**的词条，如果直接写字符串拼接（如 `t('hello_' + name)`），CLI 静态扫描是无法预知运行时变量的，这也是业界通病。
    
    为了彻底解决这个工程痛点，我们在团队内部**推行了国际化开发规范**，严禁使用字符串拼接，统一改用 **i18next 支持的模板语法（Interpolation）**。 例如统一写成：`t('welcome_message', { name: userName })`。 这样一来，对于 CLI 来说，`welcome_message` 是一个完整且固定的静态 Key，能够被 100% 完美提取。而翻译平台上的文本则是 `欢迎你，{{name}}`，交由 i18next 在运行时动态注入。这既解决了 CLI 提取率的问题，也解决了多语言语序不同的问题。”


### 追问 1：动态词条拼接，你们是怎么解决的？

- **糟糕的现状（痛点）**：以前大家喜欢用 JS 字符串拼接，比如 `const text = "当前有" + count + "个用户";`。这在中文没问题，但在英语、日语里语序完全不同，直接拼接会导致翻译出来逻辑不通、语序混乱，而且没办法处理单复数（User 还是 Users）。
    
- **你的破局方案（亮点）**：
    
    1. **语义化占位符**：废除一切代码层面的字符串拼接，统一改用带命名空间的占位符。例如：`t('user_count_tip', { count: count })`。
        
    2. **配置化映射**：在语言包里定义完整的句子骨架（中文：`当前有 {count} 个用户`，英文：`There are {count} users`）。
        
    3. **高级复数/上下文处理**：引入 `i18next` 或 `FormatJS` 的 ICU 消息语法，利用内置的 `plural` 机制（如 `{count, plural, =0 {no users} one {1 user} other {# users}}`）来动态处理单复数和性别语境。
        

### 追问 2：什么是“增量 i18n 提效”？你们具体是怎么落地的？

代码只扫描新写的（增量提取），翻译只翻新加的（增量协作），发布只发修改的（CDN解耦、无需走CI/CD）。

“增量步骤”：

1. **增量提取（Incremental Extraction）**： “我们利用 `ti18n CLI` 静态扫描源码。它不会盲目全量导出，而是通过 AST（抽象语法树）识别出当前的全部词条后，与线上已有的词条进行 **Diff 差集计算**，**精准锁定本次需求新增/修改的‘增量词条’**。”
    
2. **增量翻译与协作（Incremental Translation & Collaboration）**： “这部分增量词条会被自动同步到腾讯 ti18n 平台。翻译老师的后台**只会看到这几条新提交的文案**，无需在一本厚厚的全局字典里大海捞针，做到了**翻译层面的增量提效**。”
    
3. **增量/动态热发布（Incremental / Dynamic Deployment）**： “翻译完成后，平台一键发布。由于我们接入了 **CDN 静态化管理**，更新后的语言包直接实时推送到 CDN 节点。这使得**翻译发布与大前端代码部署（CI/CD）完全解耦**。我们不需要为了改一个错别字而重新打包整个前端工程，实现了**发布层面的增量提效**。”
