## 关于AI

在古法coding的时代，开发经常都在做一件事，就是**不断向上抽象，不断屏蔽底层**

从机器码到高级语言，从手动操作 DOM 到声明式框架，开发者越来越少关注底层执行细节，而更多关注业务逻辑和功能实现。

现在AI驱动的开发模式，就是在这个基础上又向上抽象了一层，开发者可以通过自然语言描述需求，由 AI 完成代码生成、逻辑实现，中间AI如何理解需求、如何生成代码、代码如何运行，这些过程同样可以被高度封装和屏蔽。

但这并不意味着工程复杂度消失了，而是复杂度发生了转移。

过去工程师主要关注‘如何实现代码’，未来会越来越关注：

- 需求定义
- 系统约束
- 接口契约
- 数据 schema
- 架构一致性
- AI 输出治理

因为 AI 很容易生成局部正确但全局失控的系统。

所以 AI 时代真正重要的能力，不再只是编码能力，而是定义边界、治理复杂度和约束系统行为的能力。

### 开发者在ai时代有什么用

| 旧站位      | 新站位               |
| -------- | ----------------- |
| 把需求翻译成代码 | 把模糊的问题翻译成清晰的需求    |
| 会使用某个框架  | 知道某个框架适合业务        |
| 会修某个bug  | 知道该bug暴露了哪一层的设计缺陷 |

从“生产线工人”解放出来，把更多的时间用在设计、质检上。

#### bug暴露问题例子

比如说拿AI写后台，本应该给空的数组，但是直接给了null，前端在判断是否展示列表的时候可能就是用array.length来判断，如果是null的话，访问不到length数据，就会导致js报错页面白屏。因为在AI开发的时候，大家各写各的，没有一个很细致的统一开发规范，不会发现局部代码风格不一致的问题，从这个就可以得出前后端需要一个接口契约来规避这类问题。

尤其在 AI 辅助开发场景下，如果没有约束，AI 很容易生成语义正确但类型不一致的代码，最终导致局部逻辑正常、系统协作异常。

另外这个问题也反映出前端缺少运行时数据兜底，导致异常数据直接进入渲染层并引发白屏。

所以这个 bug 不只是一个判空问题，而是接口设计、协作规范和系统容错能力共同暴露出来的问题

## **Memory**性能排查

Chrome DevTools 的 **Memory** 工具主要用来查三类问题：**内存泄漏、内存占用过大、频繁 GC 导致卡顿**。最常用的是 `Heap snapshot`。

**最常用流程：查内存泄漏**

1. 打开 DevTools  
   `F12` 或 `Cmd + Option + I` / `Ctrl + Shift + I`

2. 进入 `Memory` 面板

3. 选择 `Heap snapshot`

4. 点 `Take snapshot` 拍第一张快照

5. 在页面上执行你怀疑会泄漏的操作  
   比如打开弹窗、切换路由、上传图片、反复进入退出页面

6. 再点 `Take snapshot` 拍第二张

7. 切到 `Comparison` 对比视图  
   看哪些对象数量或内存增加了，但本来应该被释放。

重点看这些列：

```text
Constructor：对象类型
Distance：离 GC Root 的距离
Shallow size：对象自己占多少内存
Retained size：如果释放它，能连带释放多少内存
Delta：两次快照之间增加/减少了多少
```

一般排查时，最关心的是：

```text
Retained size 大
Delta 持续增加
对象本该销毁但还在
```

比如 React/Vue 页面切走后，某些组件实例、DOM 节点、数组、事件回调还在，就可能是泄漏。

**几个模式怎么选**

`Heap snapshot`：最常用，适合看“现在内存里有什么”。  
官方文档也推荐用它分析 JS 对象、DOM 节点、比较快照和查泄漏。  
来源：[Chrome DevTools Heap snapshots](https://developer.chrome.com/docs/devtools/memory-problems/heap-snapshots)

`Allocation instrumentation on timeline`：适合看“操作过程中谁在不断分配内存”。  
比如你点击按钮、滚动页面、播放动画时内存一直涨，可以用它录制过程。  
来源：[Chrome DevTools Allocation timeline](https://developer.chrome.com/docs/devtools/memory-problems/allocation-profiler)

`Allocation sampling`：开销相对小，适合长时间观察内存分配热点。

**看泄漏时的小技巧**

拍快照前可以先点一下垃圾回收按钮，也就是 Memory 面板里的小垃圾桶图标，让结果更干净。

如果看到 `Detached HTMLDivElement`、`Detached DOM tree` 之类，通常表示 DOM 已经从页面移除了，但还被 JS 引用着。常见原因是事件监听没移除、全局变量引用、闭包引用、定时器没清掉。

如果某个对象占用很大，点它后看下面的 `Retainers`。  
`Retainers` 会告诉你：**是谁还在引用它，导致它不能被回收**。这是查泄漏最关键的地方。

**一个典型泄漏例子**

```js
let cache = [];

function openPage() {
  const bigData = new Array(100000).fill('data');
  cache.push(bigData);
}
```

每次调用 `openPage()`，数据都被全局 `cache` 引用着，GC 无法回收。你在 Memory 里会看到数组数量和内存持续增长。

另一个常见例子：

```js
function mount() {
  window.addEventListener('resize', onResize);
}

function unmount() {
  // 忘了 removeEventListener
}
```

应该清理：

```js
function unmount() {
  window.removeEventListener('resize', onResize);
}
```

一句话记忆：

**Heap snapshot 看“谁还活着”，Comparison 看“谁越来越多”，Retainers 看“谁不让它死”。**
