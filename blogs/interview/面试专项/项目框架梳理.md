### 核心流程

- 用SSE来实现，引入了microsoft/fetch-event-source来建立SSE的长连接，支持 POST 请求、自定义 Header、AbortController 中断以及 SSE 数据流消费。
- 封装了一套事件驱动的状态机，用来驱动”思考/生成/完成“这些**状态UI的切换**，对内容进行增量，按照消息id和index（序列号）来累积拼装到同一条消息对象上；还可以根据消息的类型将它们放到不同的渲染对象里去渲染。

### 考点2 怎么理解事件驱动的状态机

一条消息被传入这套状态机后，会经历一系列被事件驱动的状态流转，状态要变化必须有某个触发条件

事件来了 -> 看什么类型 -> 调对应的处理函数 -> 改消息对象的状态字段 + 积累数据 -> 通知ui重渲染

### 考点3 Store 怎么设计？多模态怎么渲染？性能怎么优化？

> Store 层我会采用 Message + Block Tree 的结构设计。Message 负责维护消息级状态，例如 thinking、generating、completed，而具体内容则拆分为 Text、Image、Tool、Chart 等 Block。服务端返回事件流后，通过事件驱动更新对应 Message 和 Block 状态。
> 
> 渲染层采用 Renderer Registry 模式，根据 Block 类型动态映射到不同 Renderer 组件，实现多模态内容的统一扩展和渲染，新增一种内容类型时只需要增加新的 Renderer 即可。
> 
> 性能方面主要解决高频 Token 更新问题。我会先将 Token 写入 Buffer，再结合 requestAnimationFrame 批量更新 Store，避免每个 Token 都触发一次渲染。同时针对长会话采用虚拟列表减少 DOM 数量，对于 Markdown、代码高亮等重计算任务则采用批量解析或 Worker 异步处理，保证流式输出过程的流畅性。

### 难点+解决

#### 流式输出一些基本知识

智能体的流式输出通常基于 LLM 的 Streaming 能力实现。模型生成 Token 后，通过 SSE 实时推送给前端。对于 Agent 场景，不仅会流式返回最终答案，还会把思考过程、工具调用开始/结束、检索结果等事件统一封装成事件流。前端收到事件后**增量渲染**内容。为了避免高频 Token 导致大量渲染事件，通常会结合 Buffer 和 requestAnimationFrame 做批量更新，从而兼顾实时性和渲染性能。

#### Vue 版 Buffer + RAF

``` vue
<script setup>
import { ref } from "vue";

const message = ref("");

let buffer = [];
let scheduled = false;

function flush() {
  message.value += buffer.join("");

  buffer = [];

  scheduled = false;
}

function scheduleFlush() {
  if (scheduled) return;

  scheduled = true;

  requestAnimationFrame(flush);
}

const eventSource = new EventSource(
  "/api/chat"
);

eventSource.onmessage = (event) => {
  if (event.data === "[DONE]") {
    eventSource.close();
    return;
  }

  buffer.push(event.data);

  scheduleFlush();
};
</script>
```

> 从原理上讲，大模型本身是自回归生成的，会逐 Token 输出内容。流式输出本质上是将模型生成过程实时暴露给用户，而不是等全部内容生成完成后一次返回，这样首字延迟可以从几秒降低到几百毫秒。

> 实现上，我们会开启模型的 stream 模式，后端通过 SSE 将 Token 持续推送给前端。前端通过 ReadableStream 或 EventSource (项目直接用microsoft/fetch-event-source)持续消费数据流，并解析成事件。

> 在工程实现中，我不会每收到一个 Token 就直接更新 UI，而是会先写入 Buffer，再结合 requestAnimationFrame 或节流策略批量更新 Store，避免高频 Render 导致性能问题。

> 此外还需要处理一些边界情况，例如 TCP 拆包导致的一条 SSE 消息被拆成多段、UTF-8 字符被截断导致乱码等问题，通常会使用 buffer 和 TextDecoder 的 stream 模式解决。

> 如果是 Agent 场景，我会把流式数据抽象成 Event Stream，而不仅仅是文本流。事件可能包括 TextDelta、ToolStart、ToolEnd、ImageCreated 等，再通过 Store 更新 Block Tree，最终由不同 Renderer 组件渲染对应内容。

#### 一次性大数据如何渲染

首先要分析瓶颈是在**数据计算**、**DOM 数量**还是**浏览器布局绘制**（限制渲染频率也是一种手段）。对于普通列表场景，我会优先采用分页或虚拟列表，只渲染可视区域的数据，从根本上减少 DOM 数量。如果必须一次展示大量数据，可以结合 requestAnimationFrame 做时间切片，将渲染任务拆分到多个帧中执行，避免长任务阻塞主线程。对于 AI 场景中的长对话或日志列表，通常会结合虚拟滚动；对于超大文本会使用 Worker 做异步解析；对于海量图表数据则会采用数据采样或 WebGL 渲染。核心思路是减少单帧工作量，保证页面保持 60FPS 左右的流畅度。

#### 虚拟列表（上一条拓展）

虚拟列表的核心思想是只渲染可视区域的数据，而不是渲染全部数据。对于固定高度列表，可以通过 scrollTop 和 itemHeight 计算当前可见区域对应的 startIndex 和 endIndex，然后只渲染这一段数据。同时使用一个 phantom 元素撑开总高度，保证滚动条长度正确，再通过 translateY 将可见区域定位到正确位置。这样无论数据量是 1 万条还是 10 万条，实际 DOM 数量始终维持在几十个，从而显著降低渲染和布局开销。

#### 数据流不一定只有文字，多种格式数据流如何渲染

我会将服务端返回的数据抽象成事件流而不是字符串流。消息结构采用 Block Tree 设计，一条消息由多个 Block 组成，每个 Block 对应一种内容类型，例如 Text、Image、Chart、Code、Tool 等。前端维护统一的 Runtime Store，消费 SSE 或 WebSocket 事件后更新对应 Block。渲染层通过 Renderer Registry 动态选择对应组件进行展示。对于文本 Block 支持 Token 级流式更新，对于图片、图表、工具状态等内容则通过事件驱动更新 Block 状态。这样既支持流式输出，也支持多模态内容扩展，新增一种内容类型时只需要增加新的 Block 类型和 Renderer 即可。

#### 打字机效果
打字机效果本质上是控制文本逐步显示。对于静态文本，可以通过 setInterval 或 requestAnimationFrame 按字符追加内容实现。对于 AI 流式输出场景，通常会将服务端返回的 Token 先存入 Buffer，再通过 requestAnimationFrame 按一定速率消费 Buffer 中的内容并更新 UI，形成打字机效果。这样可以将模型输出速度和页面渲染速度解耦，同时避免高频 Token 导致大量状态更新和重渲染。实际项目中通常会结合 Buffer 和 requestAnimationFrame 实现更平滑的展示效果。
