## webpack
### 名词解释
#### Module 
webpack里的资源，每个文件都可以当作一个moudle

#### Chunk
编译完即将输出的时候，多个moudle组成一个个chunk

#### bundle
构建的输出，一个或多个chunk合并优化的结果

### 核心流程解析
初始化阶段 -》 构建阶段 -〉 生成阶段

**初始化阶段**：初始化参数，对配置文件对象、Shell参数进行解析

Webpack 会先根据 entry 找到入口模块，
然后从入口开始递归分析依赖。

在构建 module 的过程中，
会先通过 loader 把代码转译成 JS，
再解析成 AST，
分析其中的 import、require 等依赖关系，
继续递归构建所有依赖模块，
最终形成模块依赖图(Module Graph)。

之后 Webpack 会根据依赖图把多个 module 组装成 chunk，
最后将 chunk 打包生成浏览器可运行的 js、css 等静态资源。

### Webpack 解决了什么问题
- 对代码模块化，解决了全局变量污染的问题
- 可以对依赖进行管理，维护script 依赖顺序；统一了各类资源的管理
- 性能优化，比如treeshaking、Code Spliting代码分包、压缩（文件合并、压缩、减少 HTTP 请求，繁琐且易出错）、
- 解决ES6 兼容性问题（webpack可以把浏览器不能直接运行的模块化代码和各种资源打包转换成浏览器可以运行的静态资源）

### Loader 和 Plugin有什么区别
#### Loader
Loader是模块转换器，因为 Webpack 默认只能识别 JavaScript 和 JSON，Loader 可以对不同类型的模块进行转换，比如 TS、JSX、CSS、Less 等，  最终把它们转换成 Webpack 能处理的模块。

常见的loader：

- babel-loader ES5转ES6
- css（解析CSS文件）、style（把CSS注入HTML）、sass/less、postcss（自动添加CSS前缀）
- file 解析文件路径
- vue 编译vue成文件成js

#### Plugin
Plugin 本质上是基于 Webpack 生命周期 hooks 的扩展机制。

Webpack 在编译过程中会暴露很多 hooks，Plugin 可以监听这些 hooks，在**构建的不同阶段**执行**自定义逻辑**，从而扩展 Webpack 的能力。

常见plugin：

- HtmlWebpackPlugin：生成HTML文件，并自动将打包后的javaScript和CSS文件引入到HTML文件中。
- CleanWebpackPlugin：清除输出目录。
- ExtractTextWebpackPlugin：将CSS代码提取到单独的CSS文件中。
- UglifyJsWebpackPlugin：压缩JavaScript代码。

### Webpack热更新（Hot Module Replancement）

修改代码后，不用刷新页面，新代码会替代旧代码。

webpack-dev-server和浏览器之间维护了一个websocket服务。本地资源更改后，webpack会重新构建打包把新的文件存在内存里，通过WDS推送给浏览器，并附带相应的hash，让客户端和原有资源进行比较。

**用法**

1. 通过配置项`devServer.hot: true`，启用 HMR 功能。
2. 或者使用`HotModuleReplacementPlugin` HMR 插件。

### 什么是Code Splitting

Code Splitting代码分割，是一种优化技术。它允许将一个大的chunk拆分成多个小的chunk，从而实现按需加载，减少初始加载时间，并提高应用程序的性能 。

在Webpack中通过optimization.splitChunks配置项来开启代码分割

### Webpack的Tree Shaking原理

基于ES Module，在构建依赖图的时候标记没有使用的export，最后压缩的时候删掉不必要的代码

### Webpack 性能优化方法（高频）

减少打包体积、多线程并行打包、利用缓存提效(Webpack 5 引入的持久化缓存)

> 使用 `thread-loader` 或 `parallel-webpack` 可以将打包任务分配到多个进程，提高打包速度


### 长缓存、持久化缓存
文件有变化的时候才下载新版本文件，减少不必要的请求和重复加载来提升加载速度。

#### 缓存方案
1、文件名加hash，如果hash变化重新请求
2、配置缓存头Cache-Control，设置缓存时间
3、分开管理动静态资源，静态资源缓存时间长，动态短

长缓存：hash和缓存头来缓存静态资源（主要用于生产环境的资源缓存）
持久化缓存：构建过程产生的数据存在**磁盘**上，加快构建速度（主要用于开发环境中的构建优化）

### 按需加载
webpack编译的时候对动态导入【import()】的模块进行切分成chunk，然后在运行的时候动态创建script来加载这些chunk，浏览器会对这些chunk进行下载和执行。

### CommonJS 和 ES Module 的区别

commonjs在运行加载，是动态加载的，ESM是编译加载的属于静态加载，所以ESM可以支持TreeShaking（动态加载的话，不知道哪些内容是没有必要的）

|                   | CommonJS                                  | ES Module            |
| ----------------- | ----------------------------------------- | -------------------- |
| 加载时机              | 运行时加载                                     | 编译时静态加载              |
| 导入方式              | require                                   | import               |
| 导出方式              | let count = 1  <br>module.exports = count | export let count = 1 |
| 是否支持静态分析          | 不好支持                                      | 支持                   |
| 是否支持 Tree Shaking | 较难                                        | 支持                   |
| 是否支持动态加载          | 支持                                        | import() 支持          |
| 主要使用场景            | Node.js                                   | 现代前端                 |

## Vite
### Webpack 和 Vite 的区别
| 对比项          | Webpack                          | Vite                                           |
| ------------ | -------------------------------- | ---------------------------------------------- |
| 核心思想         | 基于打包的构建方式（bundle-based）          | 基于浏览器原生 ES Module的构建方式（native ESM-based）       |
| 开发模式         | 启动前先整体打包                         | 按需编译，请求时实时加载                                   |
| 冷启动速度        | 项目越大越慢                           | 非常快，大项目优势明显                                    |
| HMR 热更新      | 可能重新构建较大依赖树                      | 只更新当前模块，速度更快                                   |
| 开发环境原理       | 构建完整 dependency graph 后生成 bundle | 浏览器直接请求 ES Module（也有打包，但是只打包node_modules再给浏览器） |
| 生产构建         | 使用 Webpack 打包                    | 使用 Rollup 打包                                   |
| JS 编译工具      | Babel / Terser 等                 | esbuild（开发）、Rollup（生产）                         |
| Tree Shaking | 支持                               | 支持（Rollup 更彻底）                                 |
| 配置复杂度        | 较高                               | 较低，开箱即用                                        |
| 插件生态         | 非常成熟                             | 生态快速发展中                                        |
| 扩展能力         | 很强，适合复杂工程                        | 相对简单                                           |
| 浏览器兼容性       | 更强                               | 开发环境依赖现代浏览器 ESM                                |
| 大型项目启动       | 较慢                               | 更快                                             |
| 适合场景         | 大型复杂项目、老项目、深度定制                  | 现代前端项目、追求开发体验                                  |
| 本质区别         | 开发时先 bundle                      | 开发时不 bundle，直接使用 ESM                           |

从表里可以看出什么

 - webpack插件生态更成熟，而且提供的构建控制体系更完整，工程化能力更强；并且兼容性也更好，对CommonJS-heavy 项目支持性更好
 - vite的Tree Shaking 更彻底，配置更简单，生成代码更轻

### 为什么Vite 速度比 Webpack 快

1、**开发模式**差异（注意Webpack 和 Vite 最大区别主要体现在开发环境）

webpack先打包构建再启动项目，vite先启动，利用了现代浏览器支持ES Module的特性，在请求所需要模块时再实时编译。

2、webpack基于node.js，是毫秒级别的；vite基于esbuild，esbuild基于go开发，是纳秒级别的，所以速度上快了很多。

3、webpack热更新需要重新编译，vite只要重新请求。

### Esbuild和Rollup
#### Esbuild

是什么：一个极快的 JS/TS/JSX 转译器和打包器，用 Go 语言写的。

- Dev 模式下做**转译**—— .ts / .tsx / 现代 ES 语法，浏览器无法解析，esbuild 把它们转成浏览器能跑的代码。每个文件按需转，速度极快。
- **预构建依赖**（dep optimization）—— vite是基于ESM的，而node_modules 里很多包还是 CommonJS 格式或者拆得太碎（lodash 一个包几百个小文件），vite 启动时用 esbuild 把它们打成单个 ESM 文件，丢到 node_modules/.vite/deps/，这样可以让浏览器可以直接处理，而且减少了浏览器请求次数。

#### Rollup

是什么：是一个基于 ES Module 的打包工具，  它会构建模块依赖图，  进行 Tree Shaking、Code Splitting 等优化，  最终生成体积更小、更干净的 bundle。

##### 为什么 dev 用 esbuild、build 用 Rollup：

Rollup 的 Tree Shaking 和产物优化能力非常强。

|                 | esbuild | Rollup   |
| --------------- | ------- | -------- |
| 速度              | 极快      | 慢一些      |
| Tree-shaking 质量 | 一般      | 极好       |
| 生态成熟度           | 较新      | 非常成熟，插件多 |
| 产物质量            | OK      | 极佳，业界标准  |
